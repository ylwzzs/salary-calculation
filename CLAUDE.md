# CLAUDE.md — 项目协作规则

> 本文件每次会话自动加载，对 Claude 有约束力。**用户明确指令优先级最高**；本规则与默认行为冲突时，以本规则和用户指令为准。

---

## 第一部分：铁律（不可违背）

### 铁律 1 — 架构调整必须审批
任何涉及**架构层面**的调整，必须先报方案、经我（用户）审批通过后才能动手，不得先改后报。
"架构层面"包括但不限于：数据模型/schema、数据流与真值源、模块边界与依赖方向、技术选型（库/运行时/存储）、计算口径与业务规则、并发与事务模型、部署拓扑。

### 铁律 2 — 方案走不通时先分析根因，换方案须我同意
执行某方案遇到走不通时，**先分析并说明根因**，不得擅自更换方案继续执行。确需换方案前，必须把"为什么走不通 + 根因 + 拟换的新方案"报我，经我同意后再切换。

### 铁律 3 — 架构改动确认后，先写入架构文件，再执行
任何架构改动在我确认后，**先写入架构文件 `docs/ARCHITECTURE.md`**（记录已确认决策 + 决策过程），再进入执行；详细设计 spec 放 `docs/superpowers/specs/`、实现计划放 `docs/superpowers/plans/`。没有落架构文件的架构改动不开始编码。

### 铁律 4 — 架构与方案决策必须记录决策过程
所有涉及架构和方案的决策，记录**决策过程**（背景 → 考虑过的备选方案 → 为何选/弃 → 结论），避免时间长了重复抛出同一问题。

### 铁律 5 — 踩过的坑必须记录
所有踩过的坑（bug、误判、反复试错、环境陷阱、口径混淆等）必须记录，避免下次重犯。

## 执行机制（铁律的落地方式）

| 铁律 | 落地方式 |
|---|---|
| 1 审批 | 架构/方案以对话中我的**明确确认**（含选项选择）为准；不确定是否算"架构层面"时，默认按架构处理并报我。 |
| 2 换方案 | 执行中遇到走不通→停下→报"根因+拟换方案"→等我同意。不擅自改方案执行。 |
| 3 落文件 | **已确认架构决策统一写入 `docs/ARCHITECTURE.md`**（ADR 式）；详细设计 spec 写 `docs/superpowers/specs/`；实现计划写 `docs/superpowers/plans/`。ARCHITECTURE.md 经我复审后才进入 writing-plans 与编码。 |
| 4 决策记录 | **已确认架构决策统一记入 `docs/ARCHITECTURE.md`**（背景/备选/理由/影响）；踩坑与跨期结论同步到记忆系统（`memory/`，type=feedback/project）。 |
| 5 踩坑记录 | 写入记忆系统（`memory/`，type=feedback，附 **Why** + **How to apply**）。 |
